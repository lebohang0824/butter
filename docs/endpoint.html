Endpoint

Declares a synchronous HTTP transport endpoint — route, method, parameters, response schemas, execution actions, and return mappings. The complete API contract in a single self-contained block.

An endpoint block describes a complete HTTP API contract: the route and HTTP method, typed input parameters, response schemas with typed fields, sequential execution actions, and return mappings that connect status codes to response payloads. Everything the AI agent needs to implement the endpoint is packaged within the block boundaries.

Like features, endpoints are strictly self-contained. All params, responses, actions, and returns are scoped within the endpoint block. No cross-endpoint references, no shared state. This isolation enables selective compilation — extracting a single endpoint produces a zero-noise payload for AI consumption.

Syntax

An endpoint is declared at the top level of a .butter file, directly under the app / product header:

app MyApp
description "My API"
version "1.0.0"

endpoint ProcessOrder
  description "Processes an order and returns a summary"
  version "1.0.0"
  route "/api/orders"
  method "POST"

  params
    param order_token
      type string
      required true

  responses
    response OrderSuccess
      field order_id
        type int
      field total
        type float

  actions
    action "Validate the order token"
    action "Calculate order total"

  returns
    return 201 OrderSuccess
    return 400 "Invalid order token" | if "Order token validation fails"

Rules

  • Each endpoint must have a unique name within the spec — duplicate endpoint names produce a semantic error.
  • The endpoint name is an identifier: letters, digits, underscores, and hyphens are allowed.
  • The content under an endpoint must be indented.
  • Valid keywords inside an endpoint: description, version, route, method, params, responses, actions, returns.
  • route and method are required — an endpoint without them produces a semantic error.
Note: The feature primitive covers domain logic without HTTP transport concerns. Use feature for business capabilities and endpoint for API contracts. See the Feature documentation for details.

HTTP Fields

Endpoints declare two HTTP-specific fields that features do not have:

route

The URL path for the endpoint. Must be a quoted string starting with /:

route "/api/checkout/orders"
route "/api/users/:id/profile"
route "/api/products"

method

The HTTP method. Must be a quoted string matching one of the standard HTTP methods:

method "GET"
method "POST"
method "PUT"
method "DELETE"
method "PATCH"

Semantic Validation

  • An endpoint without a route produces the error: endpoint "..." is missing a route
  • An endpoint without a method produces the error: endpoint "..." is missing a method
  • Both are hard errors — compilation halts and no output is generated.

Sub-blocks

An endpoint contains four optional sub-blocks:

  route "/api/orders"
  method "POST"

  params
    param customer_id
      type string
      required true

  responses
    response OrderCreated
      field order_id
        type int
      field status

  actions
    action "Validate customer exists"

  returns
    return 201 OrderCreated
    return 400 "Invalid customer ID"

Params

The params block inside an endpoint declares the input parameters the endpoint accepts. The syntax and field types are identical to feature params — see the Feature Params documentation for the full reference on types, defaults, validate rules, and length constraints.

params
  param checkout_token
    type string
    required true
  param coupon_code
    type string
    required false
    default "WELCOME10"
  param dry_run
    type bool
    required false
    default false

Endpoint Param Conventions

While feature params are typically PascalCase identifiers (e.g. Title, OrderID), endpoint params often use snake_case to match HTTP parameter naming conventions (e.g. checkout_token, order_id). Both styles are valid — choose the convention that matches your API design.

Semantic Rules

  • Parameter names must be unique within an endpoint.
  • Default values must match declared types.
  • Enum defaults must exist in the defined list.
  • validate requires type int or type float.
  • length and validate are mutually exclusive.

Responses

The responses block declares named response schemas. Each response has a name and a list of typed fields. Response names are referenced by return statements to connect status codes to their payload schemas.

responses
  response OrderSuccessPayload
    field order_id
      type int
    field total_amount
      type float
    field currency
    field line_items
      type array[string]

  response ValidationError
    field error
    field details
    field field_name

Response Name Conventions

Response names are identifiers — use PascalCase or descriptive names that clearly communicate the payload's purpose:

  • OrderSuccessPayload — descriptive, clearly a success response
  • ValidationError — communicates the error type
  • UserProfile — domain-specific, clear intent

Semantic Rules

  • Response names referenced in return statements must be defined in the responses block. Unresolved references produce a semantic error.
  • Response names within a single endpoint must be unique.

Fields

Each response contains one or more field entries. Fields declare the shape of the response payload that the AI agent should produce.

response OrderSuccess
  field order_id
    type int
  field total_amount
    type float
  field customer_name
  field line_items
    type array[string]

Field Syntax

  • Name: The identifier after field (e.g. order_id).
  • Type: Optional type keyword. Defaults to string if omitted.
  • Nesting: Fields can contain sub-fields for complex nested structures.

Nested Fields

Fields support nesting for complex response structures. Indent additional field entries under a parent field:

response InvoiceLineItem
  field sku
  field description
  field unitPrice
    type float
  field quantity
    type int

response OrderSuccess
  field orderId
  field total_amount
    type float
  field currency
  field lineItems
    type array[InvoiceLineItem]

Compiled Output

Fields compile to a JSON array of objects. Each field has name, type, and optionally sub_fields:

{
  "name": "OrderSuccess",
  "fields": [
    { "name": "orderId", "type": "string" },
    { "name": "total_amount", "type": "float" },
    { "name": "currency", "type": "string" },
    { "name": "lineItems", "type": "array[InvoiceLineItem]" }
  ]
}

Actions

The actions block inside an endpoint declares the sequential execution steps for processing the request. The syntax and execution model are identical to feature actions — see the Feature Actions documentation for the full reference on action statements, conditions, and the sequential execution model.

actions
  action "Verify checkout token is valid and active"
  action "Calculate final prices using coupon_code" | if "coupon_code exists"
  action "Authorize payment transaction via external provider" | unless "dry_run == true"
  action "Commit transaction details and reserve inventory" | unless "dry_run == true"

Endpoint Action Conventions

Endpoint actions typically describe request processing steps: validation, business logic, external calls, and side effects. They follow the same sequential, deterministic model as feature actions — one step at a time, in declared order.

Returns

The returns block maps HTTP status codes to response schemas or string payloads. Each return statement connects a status code to either a named response (defined in responses) or a quoted string literal, with an optional condition.

returns
  return 201 OrderSuccessPayload
  return 200 OrderSuccessPayload | if "Dry run validation succeeded"
  return 400 "Invalid or expired checkout token" | if "Checkout token validation fails"
  return 402 "Payment authorization failed" | unless "Payment gateway processes transaction"
  return 500 "External payment provider connection times out"
  return 404

Return Statement Format

return <status_code> [<payload>] [| if/unless "<condition>"]
ComponentRequiredDescription
status_codeYesHTTP status code (integer): 200, 201, 400, 404, 500, etc.
payloadNoEither a response name (references responses block) or a quoted string literal
ConditionNoOptional | if or | unless with a quoted expression

Payload Types

Response Reference

Reference a named response from the responses block. The AI agent uses the response schema to structure the output:

return 201 OrderSuccessPayload

String Literal

A quoted string for simple error messages or status text:

return 400 "Invalid or expired checkout token"
return 500 "Internal processing error"

Status Code Only

A bare status code with no payload — for responses with no body:

return 204
return 404

Conditions on Returns

Return statements support | if and | unless conditions (but not | when or | while). The condition determines which return path is taken based on the request context:

return 201 OrderSuccess | if "Payment succeeds"
return 200 "Dry run completed" | if "dry_run == true"
return 400 "Validation failed" | unless "Input is valid"
return 500 "Timeout" | if "External provider connection times out"

Semantic Rules

  • Response names referenced in return statements must be defined in the responses block. If a return references OrderSuccessPayload but no such response exists, compilation fails.
  • Status codes must be valid HTTP status codes (integers).
  • Return conditions only support if and unless — when and while are not valid for return statements.

Compiled Output

Each return compiles to a JSON object with status_code, optional payload, optional payload_is_string (true for string literals), and optional condition:

{
  "status_code": 201,
  "payload": "OrderSuccessPayload"
},
{
  "status_code": 400,
  "payload": "Invalid or expired checkout token",
  "payload_is_string": true,
  "condition": {
    "type": "if",
    "expression": "Checkout token validation fails"
  }
},
{
  "status_code": 404
}

Scope Isolation

Every endpoint in Butter is strictly self-contained. Parameters, responses, actions, and returns within an endpoint have no knowledge of or dependency on other endpoints or features. This design ensures:

  • No cross-references — An endpoint cannot reference params, responses, or actions from another endpoint.
  • No shared state — Each endpoint is an independent API contract.
  • Independent compilation — Each endpoint can be isolated and compiled on its own.
  • Zero noise — Extracting a single endpoint produces output with zero references to other endpoints.

This isolation is what makes selective compilation possible. Because each endpoint is a complete, self-contained unit, the compiler can extract it without resolving external dependencies.

endpoint ProcessOrder
  route "/api/orders"
  method "POST"
  params
    param token
      type string
      required true
  actions
    action "Validate token"

endpoint GetOrderStatus
  route "/api/orders/:id"
  method "GET"
  params
    param order_id
      type string
      required true
  actions
    action "Look up order by ID"

Compiling with --only endpoint:ProcessOrder produces output containing only ProcessOrder — zero mention of GetOrderStatus.

Selective Compilation

Butter's CLI supports extracting specific endpoints via the --only and --exclude flags:

Extract a Single Endpoint

butter compile api.butter --only endpoint:ProcessOrder -f json

Extract Multiple Endpoints

butter compile api.butter --only endpoint:ProcessOrder,endpoint:GetOrderStatus -f json

Mix Features and Endpoints

butter compile api.butter --only endpoint:ProcessOrder,feature:ApplyCoupon -f json

Exclude an Endpoint

butter compile api.butter --exclude endpoint:HealthCheck -f json

How It Works

  1. The compiler parses the full .butter file into a complete AST.
  2. Semantic analysis runs on the full AST (all endpoints are validated).
  3. The filter engine prunes the AST — un-matched endpoint branches are dropped.
  4. The global context (app name, description, version) is always preserved.
  5. The isolated sub-graph is serialised to the requested format.
Error handling: If --only references an endpoint name that does not exist in the spec, compilation fails with an error. If --exclude references a non-existent endpoint, it also fails. The compiler never silently ignores typos.

Human-to-AI Handoff

Selective compilation is designed for the prompt-stacking workflow:

  1. Write the endpoint spec inside your .butter file.
  2. Extract with butter compile api.butter --only endpoint:ProcessOrder -f json -o prompt.json.
  3. Paste the JSON into your AI chat alongside instructions:
Implement the code for the following Butter contract definition.
Use Go with the Gin framework. Ensure every listed action is a
distinct function step and match the return criteria exactly.

[Paste JSON here]

Compiled Output

When compiled to JSON, an endpoint produces a complete API contract with route, method, params, responses, actions, and returns:

endpoint ProcessOrder
  description "Validates checkout items and processes the transaction"
  version "1.0.0"
  route "/api/checkout/orders"
  method "POST"

  params
    param checkout_token
      type string
      required true
    param coupon_code
      type string
      default "WELCOME10"
    param dry_run
      type bool
      default false

  responses
    response OrderSuccessPayload
      field order_id
        type int
      field total_amount
        type float
      field currency

  actions
    action "Validate checkout token is valid and active"
    action "Calculate final prices using coupon_code" | if "coupon_code exists"
    action "Authorize payment transaction" | unless "dry_run == true"

  returns
    return 201 OrderSuccessPayload
    return 200 "Dry run completed" | if "dry_run == true"
    return 400 "Invalid checkout token"
    return 500 "Payment provider timeout"

app TaskAPI
description "Task management API"
version "1.0.0"

endpoint ListTasks
  route "/api/tasks"
  method "GET"
  params
    param status
      type enum["all", "active", "completed"]
      default "all"
    param limit
      type int
      default 50
      validate ">=1"
      validate "<=200"
  responses
    response TaskList

        type array[Task]
      field total_count
        type int
  actions
    action "Query task repository with filters"

endpoint CreateTask
  route "/api/tasks"
  method "POST"
  params
    param title
      type string
      required true
    param priority
      type enum["low", "medium", "high"]
      default "medium"
  responses
    response TaskCreated
      field id
        type int
      field title
      field priority
      field created_at
  actions
    action "Validate title is not empty"
    action "Assign unique identifier"
    action "Set creation timestamp"
  returns
    return 201 TaskCreated
    return 400 "Title is required" | if "title is empty"

endpoint DeleteTask
  route "/api/tasks/:id"
  method "DELETE"
  params
    param id
      type string
      required true
  actions
    action "Verify task exists"
    action "Remove task from storage"
  returns
    return 204
    return 404 "Task not found" | if "Task does not exist"
