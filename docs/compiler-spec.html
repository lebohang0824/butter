compiler.butter

Butter is a specification language for AI intent — and the compiler itself is described in its own language. This page breaks down every line, showing you how to write specs that guide AI agents toward deterministic, one-shot results.

The compiler.butter file at the project root is a self-describing specification. It declares the Butter compiler using the same DSL it compiles — every param, action, and condition maps directly to a real stage in the compiler pipeline. This is Butter's philosophy in action: a structured spec that an AI agent could follow to rebuild the compiler from scratch.

By studying this file you learn how to write your own .butter specifications for AI agents: the same product/app header, feature blocks, params with typed fields and constraints, and actions with if/unless/when/while conditions apply to any project you describe. When compiled to JSON and fed to an AI agent, the agent produces implementations that match up to 100% of expected results in a single shot.

The compiler.butter File

Below is the complete source. Every element is explained in the sections that follow.

# Butter Project Documentation
product ButterCompiler
description "A high-performance, indentation-aware DSL compiler
  that converts clean .butter specifications into structured
  JSON or YAML."
version "1.10.0"

feature CompilationPipeline
  description "The strictly sequential end-to-end execution
    flow of the Butter compiler."
  version "1.10.0"

  params
    param InputFile
      type string
      required true
    param OutputFormat
      type enum["json", "yaml"]
      default "json"
    param CheckOnly
      type bool
      default false

  actions
    # Stage 1: Lexical Analysis (The Off-side Rule)
    action "Read file sequentially and track whitespace
      with a LIFO indentation stack"
      enforce "The lexer must maintain an indentation stack
        to track scope boundaries"
      enforce "Mismatched indentation must halt compilation
        with a meaningful error"
    action "Emit INDENT token when leading whitespace
      exceeds stack top"
    action "Emit DEDENT tokens when leading whitespace
      is less than stack top"
    action "Emit NEWLINE, IDENTIFIER, STRING, PIPE,
      EOF, and ERROR tokens"
    action "Halt compilation with exit code 1"
      | when "Indentation mismatch is detected"
    # Stage 2: Recursive Descent Parsing
    action "Consume token stream to construct
      Abstract Syntax Tree (AST)"
    action "Map root tokens to AppSpec struct
      (app/product, description, version)"
    action "Map nested blocks to FeatureSpec structs"
    action "Map parameter definitions to ParamSpec structs
      (string, int, float, bool, enum)"
    action "Map execution steps to ActionSpec and
      ConditionSpec structs (if, unless, when, while)"
    action "Halt compilation with exit code 1"
      | when "Syntax validation fails"
    # Stage 3: Semantic Analysis
    action "Check AST for duplicate feature names"
    action "Check AST for duplicate parameter names
      within features"
    action "Verify default values match their declared types"
    action "Verify default values for enum types
      exist in the defined list"
    action "Enforce mutual exclusion between
      'length' and 'validate' constraints"
    action "Emit warning but continue"
      | if "A parameter is marked required but has a default"
    action "Halt compilation with exit code 1"
      | when "Any semantic error is found"
    # Stage 4: Output Generation
    action "Skip output generation and exit with code 0"
      | if "CheckOnly == true"
    action "Look up the registered output extension
      by name from the extension registry"
      | unless "CheckOnly == true"
    action "Dispatch serialization to the extension's
      Serialize method"
      | unless "CheckOnly == true"
    action "Write serialized bytes to output path
      and exit with code 0"
      | unless "CheckOnly == true"

Parameters Explained

The params block inside a feature declares the inputs that feature accepts. Each param is a named variable with a type and optional constraints.

InputFile type: string, required: true

The path to the source .butter file. This is the only required parameter — without it the compiler has nothing to read. In the CLI this maps to the positional argument butter compile <file>.

OutputFormat type: enum["json", "yaml"], default: "json"

Controls the serialization format. The compiler maintains a registry of output.Extension implementations, each providing a Name() and Serialize() method. Built-in extensions include JSON and YAML; third-party formats can be added by implementing the Extension interface and registering via output.Register(). Maps to the --format / -f CLI flag.

CheckOnly type: bool, default: false

When true, the compiler validates syntax and semantics but skips file output. The exit code is 0 on success, 1 on failure. This is the --check flag — useful for CI pipelines and editor integration (the VS Code extension calls butter compile --check on every save).

Param Field Reference

Every param in a Butter spec supports these fields:

FieldPurposeExample
typeData type: string, int, float, bool, or enum[...]type float
requiredWhether the parameter must be provided (true or false)required true
defaultFallback value when the parameter is omitteddefault "json"
validateNumeric comparison rule for int/float params. Mutually exclusive with length.validate ">=1"
lengthExact digit length for int/float params. Mutually exclusive with validate.length 13
Note: If a param has no explicit type, it defaults to string. If required is not set, it defaults to false. A parameter that is both required: true and has a default triggers a warning — the default is never used.

Stage 1: Lexical Analysis (The Off-side Rule)

The lexer reads the input file character by character and produces a stream of tokens. Because Butter uses indentation for structure (inspired by Python), the lexer maintains a LIFO indentation stack.

#ActionIntention
1 Read file sequentially and track whitespace with a LIFO indentation stack The lexer walks the file from start to finish. Every time it encounters a newline, it measures the leading whitespace of the next line and pushes or pops the indentation stack accordingly. This is the foundation of Butter's scoping — no braces, no begin/end, just indentation.
2 Emit INDENT token when leading whitespace exceeds stack top When a line is indented deeper than the current level, the lexer emits an INDENT token and pushes the new depth. This signals the parser to enter a nested block (e.g. inside a feature or params).
3 Emit DEDENT tokens when leading whitespace is less than stack top When a line is at a shallower indentation, the lexer pops levels from the stack and emits a DEDENT token for each. This closes the open blocks. If no matching indentation level is found, it's an error.
4 Emit NEWLINE, IDENTIFIER, STRING, PIPE, EOF, and ERROR tokens These are the remaining token types. NEWLINE separates lines. IDENTIFIER captures keywords (feature, param, type, etc.) and names. STRING captures quoted values. PIPE (|) separates actions from their conditions. EOF marks end of file. ERROR reports unexpected characters.
5 Halt compilation with exit code 1 | when "Indentation mismatch is detected" If the indentation stack can't find a matching level for a line's indentation, the lexer stops and returns an error. This catches inconsistent indentation (e.g. mixing 2-space and 4-space blocks).
Writing tip: Always use consistent indentation — either spaces or tabs, never both. The lexer normalises tabs to 4 spaces, but mixing causes hard-to-debug mismatch errors.

Stage 2: Recursive Descent Parsing

The parser consumes the token stream produced by the lexer and constructs an Abstract Syntax Tree (AST). It is a hand-written recursive descent parser — no parser generators, no grammar files.

#ActionIntention
6 Consume token stream to construct Abstract Syntax Tree (AST) The parser's main loop repeatedly reads tokens and builds structured Go types (AppSpec, FeatureSpec, ParamSpec, ActionSpec). Each INDENT/DEDENT pair opens/closes a scope, mirroring the indentation structure directly in the tree.
7 Map root tokens to AppSpec struct (app/product, description, version) At the top level, the parser expects either app or product (they are aliases) followed by a name, then optionally description and version. These become the root AppSpec node — the outermost container of your spec.
8 Map nested blocks to FeatureSpec structs Every feature keyword starts a new FeatureSpec. The feature name becomes the Name field, and the indented block under it contains its description, version, params, and actions. Features are the primary organisational unit in Butter — think of them as modules, APIs, or sub-systems.
9 Map parameter definitions to ParamSpec structs Inside a params block, each param becomes a ParamSpec with fields like type, required, default, validate, and length. The parser validates field names and their values at this stage.
10 Map execution steps to ActionSpec and ConditionSpec structs Inside an actions block, each action becomes an ActionSpec with a statement string. Zero or more enforce strings may appear as indented children, specifying conditions that must hold for success. If the action has a | if, | unless, | when, or | while suffix, a ConditionSpec is attached with the condition type and expression.
11 Halt compilation with exit code 1 | when "Syntax validation fails" If any token sequence doesn't match the expected grammar (e.g. a param with no name, or an unexpected keyword), the parser returns an error with the exact line number and a human-readable message.
Writing tip: Every feature block must be indented under the app/product header. Every params and actions block must be indented under its feature. Every param must be indented under params. Consistent indentation = correct parsing.

Stage 3: Semantic Analysis

After the AST is built, a dedicated semantic analysis pass checks logical rules that the parser cannot enforce. Unlike syntax errors, semantic errors are about meaning and consistency.

#ActionIntention
12 Check AST for duplicate feature names Two features cannot share the same name. If a duplicate is found, the error includes the line number of both the duplicate and the first definition. This prevents ambiguous references.
13 Check AST for duplicate parameter names within features Within a single feature, parameter names must be unique. Duplicate params are flagged with their first-definition line for easy correction.
14 Verify default values match their declared types A default of "hello" on an int parameter is a type mismatch. The analyser checks that int defaults parse as integers, float as numbers, and bool as true or false. String and enum types accept any string value.
15 Verify default values for enum types exist in the defined list If a parameter has type enum["a", "b", "c"] and default "d", the semantic analyser rejects it because "d" is not in the enum list. This guarantees that defaults are always valid choices.
16 Enforce mutual exclusion between 'length' and 'validate' constraints A parameter cannot have both a length constraint and a validate rule — they are conflicting approaches to validation. If both are present, the analyser reports an error. (The parser also checks this earlier as a safety net.)
17 Emit warning but continue | if "A parameter is marked required but has a default" A required parameter with a default is contradictory: if the value must always be provided, the default is never used. This is a warning, not an error — the output is still generated. It's the only semantic warning; all others block compilation.
18 Halt compilation with exit code 1 | when "Any semantic error is found" If any semantic check produces an error (not a warning), the compiler exits with code 1 and no output is written. This ensures that only semantically valid specs produce output.
Writing tip: Run butter compile --check yourfile.butter to catch both syntax and semantic errors before generating output. This is especially useful in CI pipelines and editor save hooks.

Stage 4: Output Generation

The final stage serialises the validated AST to disk. No semantic analysis runs here — by this point the AST is guaranteed to be valid.

#ActionIntention
19 Skip output generation and exit with code 0 | if "CheckOnly == true" When the --check flag is set, the compiler validates everything but writes nothing. Exit code 0 means the file is valid. This is the path used by the VS Code extension for on-save linting.
20 Look up the registered output extension by name from the extension registry | unless "CheckOnly == true" The --format value is used to look up an output.Extension in the global registry (registered via init()). If no extension matches, compilation halts with an error listing all registered formats.
21 Dispatch serialization to the extension's Serialize method | unless "CheckOnly == true" The selected extension's Serialize(*ast.AppSpec) method produces the output bytes. JSON and YAML are built-in extensions; third-party formats can be added by implementing the Extension interface.
22 Parse optional enforce fields as indented child blocks under actions Actions may have zero or more enforce strings indented beneath them, specifying conditions that must hold for the action to succeed. The parser appends each to the action's Enforce slice.
23 Write serialized bytes to output path and exit with code 0 | unless "CheckOnly == true" The serialised output is written to the target file (default: <input>.json or <input>.<ext>, customisable with -o). The compiler then exits with code 0, signalling success.

Writing Your Own Butter Specs

The compiler.butter file follows the same rules as any Butter spec. Here is a checklist you can use for your own files:

ElementRuleExample
Root declaration Start with app or product followed by a name app MyApp
Description Optional quoted string at root or feature level description "My app does X"
Version Optional quoted version string version "2.0.0"
Features Indented block under root, one feature per line feature Login then indented content
Parameters params block with indented param entries param Username then indented fields
Param fields Supported: type, required, default, validate, length type int, required true
Actions actions block with indented action entries action "Do something"
Conditions Append | if, | unless, | when, or | while with a quoted expression | if "x > 10"
Comments Lines starting with # are ignored # This is a comment
Indentation Must be consistent — spaces or tabs, not mixed. Tabs normalize to 4 spaces. Use 2 or 4 spaces consistently

Quick-start template

Copy this to get started with your own Butter spec:

# My Project
app MyProject
description "What my project does"
version "1.0.0"

feature MyFeature
  description "What this feature does"
  version "1.0.0"

  params
    param MyInput
      type string
      required true
    param MyOption
      type enum["a", "b", "c"]
      default "a"

  actions
    action "First step of the feature"
    action "Conditional step" | if "MyInput is set"
    action "Reactive step" | when "Some event occurs"

Compile it: butter compile myproject.butter
