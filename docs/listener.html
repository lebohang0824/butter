Listener

Declares an asynchronous message consumer — topic, parameters, execution actions, and return state mappings. The complete message processing contract in a single self-contained block.

A listener block describes a complete asynchronous message processing contract: the topic or queue to consume from, typed input parameters, sequential execution actions, and return state mappings that define how messages are acknowledged or rejected. Everything the AI agent needs to implement the listener is packaged within the block boundaries.

Like features and endpoints, listeners are strictly self-contained. All params, actions, and returns are scoped within the listener block. No cross-listener references, no shared state. This isolation enables selective compilation — extracting a single listener produces a zero-noise payload for AI consumption.

Syntax

A listener is declared at the top level of a .butter file, directly under the app / product header:

app MyApp
description "My Async API"
version "1.0.0"

listener ProcessOrderEvent
  description "Processes incoming order events"
  version "1.0.0"
  topic "orders.process"

  params
    param OrderId
      type string
      required true

  actions
    action "Validate order data"
    action "Process payment"

  returns
    return ack | if "Order processed successfully"
    return nack | if "Invalid order data"
    return retry | if "Payment gateway timeout"
    return dlq | if "Permanent processing failure"

Rules

  • Each listener must have a unique name within the spec — duplicate listener names produce a semantic error.
  • The listener name is an identifier: letters, digits, underscores, and hyphens are allowed.
  • The content under a listener must be indented.
  • Valid keywords inside a listener: description, version, topic, params, actions, returns.
  • topic is required — a listener without a topic produces a semantic error.
Note: The feature primitive covers domain logic without transport concerns. The endpoint primitive handles synchronous HTTP. Use listener for asynchronous message processing with Kafka, RabbitMQ, SQS, or similar systems.

Message Fields

Listeners declare one message-specific field that features and endpoints do not have:

topic

The message topic or queue name to consume from. Must be a quoted string:

topic "orders.process"
topic "users.activity"
topic "payments.completed"
topic "notifications.email"

Semantic Validation

  • A listener without a topic produces the error: listener "..." is missing required 'topic'
  • This is a hard error — compilation halts and no output is generated.

Params

The params block inside a listener declares the input parameters extracted from the message. The syntax and field types are identical to feature and endpoint params — see the Feature Params documentation for the full reference on types, defaults, validate rules, and length constraints.

params
  param MessageId
    type string
    required true
  param UserId
    type string
    required true
  param Payload
    type string
  param Priority
    type int
    default 0

Listener Param Conventions

Listener params typically represent message fields. Use CamelCase consistently (for example, OrderId and EventType) to match Butter's documentation examples.

Semantic Rules

  • Parameter names must be unique within a listener.
  • Default values must match declared types.
  • Enum defaults must exist in the defined list.
  • validate requires type int or type float.
  • length and validate are mutually exclusive.

Actions

The actions block inside a listener declares the sequential execution steps for processing the message. The syntax and execution model are identical to feature and endpoint actions — see the Feature Actions documentation for the full reference on action statements, conditions, and the sequential execution model.

actions
  action "Validate message schema"
  action "Extract order data from payload"
  action "Process payment transaction"
  action "Update inventory counts"
  action "Send confirmation notification"

Listener Action Conventions

Listener actions typically describe message processing steps: validation, data extraction, business logic, external calls, and side effects. They follow the same sequential, deterministic model as feature and endpoint actions — one step at a time, in declared order.

Returns

The returns block maps message processing outcomes to acknowledgment states. Each return statement specifies a message state (ack, nack, retry, or dlq) with an optional condition.

returns
  return ack | if "Message processed successfully"
  return nack | if "Invalid message format"
  return retry | if "Temporary processing failure"
  return dlq | if "Permanent processing failure"

Return Statement Format

return <state> [| if/unless "<condition>"]
ComponentRequiredDescription
stateYesMessage processing state: ack, nack, retry, or dlq
ConditionNoOptional | if or | unless with a quoted expression

Conditions on Returns

Return statements support | if and | unless conditions (but not | when or | while). The condition determines which return state is taken based on the processing context:

return ack | if "Processing succeeds"
return retry | unless "Max retries exceeded"
return dlq | if "Message is malformed"

Semantic Rules

  • Return states must be one of: ack, nack, retry, dlq. Invalid states produce a semantic error.
  • Return conditions only support if and unless — when and while are not valid for return statements.

Message States

Listener returns use message states instead of HTTP status codes. Each state tells the message broker how to handle the message after processing:

StateDescriptionTypical Use Case
ackAcknowledge — message is successfully processed and removed from the queueSuccessful processing, valid data, completed workflow
nackNegative acknowledgment — message is rejected and may be requeued or discardedInvalid message format, business rule violation, permanently invalid data
retryRetry — message will be reprocessed later (often with backoff)Temporary failures, rate limiting, transient errors, service unavailability
dlqDead letter queue — message is moved to a dead letter queue for manual inspectionPermanent failures, unprocessable messages, messages that exceed retry limits

When to Use Each State

returns
  return ack | if "Order processed and confirmed"
  return nack | if "Order data is invalid or malformed"
  return retry | if "Payment gateway is temporarily unavailable"
  return dlq | if "Order exceeds maximum retry attempts"

Scope Isolation

Every listener in Butter is strictly self-contained. Parameters, actions, and returns within a listener have no knowledge of or dependency on other listeners, endpoints, or features. This design ensures:

  • No cross-references — A listener cannot reference params, actions, or returns from another listener.
  • No shared state — Each listener is an independent message processing contract.
  • Independent compilation — Each listener can be isolated and compiled on its own.
  • Zero noise — Extracting a single listener produces output with zero references to other listeners.

This isolation is what makes selective compilation possible. Because each listener is a complete, self-contained unit, the compiler can extract it without resolving external dependencies.

listener ProcessOrder
  topic "orders.process"
  params
    param OrderId
      type string
      required true
  actions
    action "Process order"

listener TrackUserActivity
  topic "users.activity"
  params
    param UserId
      type string
      required true
  actions
    action "Log activity"

Compiling with --only listener:ProcessOrder produces output containing only ProcessOrder — zero mention of TrackUserActivity.

Selective Compilation

Butter's CLI supports extracting specific listeners via the --only and --exclude flags:

Extract a Single Listener

butter compile events.butter --only listener:ProcessOrder -f json

Extract Multiple Listeners

butter compile events.butter --only listener:ProcessOrder,listener:TrackActivity -f json

Mix Block Types

butter compile events.butter --only listener:ProcessOrder,endpoint:HealthCheck -f json

Exclude a Listener

butter compile events.butter --exclude listener:DebugListener -f json

How It Works

  1. The compiler parses the full .butter file into a complete AST.
  2. Semantic analysis runs on the full AST (all listeners are validated).
  3. The filter engine prunes the AST — un-matched listener branches are dropped.
  4. The global context (app name, description, version) is always preserved.
  5. The isolated sub-graph is serialised to the requested format.
Error handling: If --only references a listener name that does not exist in the spec, compilation fails with an error. If --exclude references a non-existent listener, it also fails. The compiler never silently ignores typos.

Human-to-AI Handoff

Selective compilation is designed for the prompt-stacking workflow:

  1. Write the listener spec inside your .butter file.
  2. Extract with butter compile events.butter --only listener:ProcessOrder -f json -o prompt.json.
  3. Paste the JSON into your AI chat alongside instructions:
Implement the code for the following Butter listener contract definition.
Use Go with the Kafka consumer library. Ensure every listed action is a
distinct function step and match the return states exactly.

[Paste JSON here]
