Writing Output Extensions

Add support for any output format by implementing one Go interface. No changes to the compiler core required.

Butter's output layer is fully pluggable. The built-in JSON and YAML serializers are just two implementations of a simple Extension interface. Anyone can write a new extension — for TOML, XML, Protobuf, Markdown, HTML, or any custom format — and plug it into the compiler with a single import. The JSON output is what you feed to AI agents; adding more output formats means more ways to communicate intent.

This guide walks through the entire process, from the interface contract to publishing your extension.

The Extension Interface

The contract lives in pkg/output/extension.go. Three methods to implement:

package output

import "butter/pkg/ast"

type Extension interface {
    // Name returns the format identifier used with --format / -f (e.g. "toml").
    Name() string

    // FileExtension returns the output file extension including the dot (e.g. ".toml").
    FileExtension() string

    // Serialize converts the validated AST into the target format and returns bytes.
    Serialize(spec *ast.AppSpec) ([]byte, error)
}

That's all. Serialize receives a validated *ast.AppSpec — the same typed AST that JSON and YAML serialise. You have full access to:

  • spec.App — The app/product name
  • spec.Description — Root description string
  • spec.Version — Root version string
  • spec.Features — Slice of FeatureSpec, each containing Name, Description, Version, Params ([]ParamSpec), and Actions ([]ActionSpec)

Every field has json and yaml struct tags for standard serialisation, but you can walk the struct tree manually for complete control over the output format.

Walkthrough: Writing a TOML Extension

This step-by-step example adds TOML output support. The finished result is one Go file and about 40 lines of code.

0. The Extension Described in Butter

Before we write Go code, let's write a Butter spec that describes the TOML extension itself. This is the same pattern used by compiler.butter (the compiler's self-description) and butter-extension.butter (the VS Code extension's self-description). We use Butter to define the extension's behaviour:

# TOML Output Extension for Butter
product TomlExtension
description "Butter output extension that serializes AppSpec AST into TOML format using github.com/BurntSushi/toml"
version "1.0.0"

feature CoreSerialization
  description "Top-level AppSpec struct serialization."
  version "1.0.0"

  params
    param Library
      type string
      default "BurntSushi/toml"
    param EncoderType
      type string
      default "toml.NewEncoder"

  actions
    action "Encode spec.App as the app top-level key"
    action "Encode spec.Description as the description top-level key"
    action "Encode spec.Version as the version top-level key"
    action "Encode spec.Features as the [[features]] array of tables"

feature FeatureSerialization
  description "Serialization of each FeatureSpec into a TOML table."
  version "1.0.0"

  params
    param ParamsAsArray
      type bool
      default true
    param ActionsAsArray
      type bool
      default true

  actions
    action "Encode feature.Name as the name key"
    action "Encode feature.Description as the description key"
    action "Encode feature.Version as the version key"
    action "Encode feature.Params as [[features.params]] sub-array"
    action "Encode feature.Actions as [[features.actions]] sub-array"

feature ParamSerialization
  description "Serialization of each ParamSpec into a TOML table."
  version "1.0.0"

  params
    param ValidateAsArray
      type bool
      default true

  actions
    action "Encode param.Name as the name key"
    action "Encode param.Type as the type key"
    action "Encode param.Required as the required bool key"
    action "Encode param.Default as the default key with inferred TOML type"
    action "Encode param.Validate as a string array" | if "len(param.Validate) > 0"

feature ActionSerialization
  description "Serialization of each ActionSpec and its optional ConditionSpec."
  version "1.0.0"

  params
    param ConditionTableKey
      type string
      default "condition"

  actions
    action "Encode action.Statement as the statement key"
    action "Encode action.Condition.Type as condition.type" | when "action.Condition != nil"
    action "Encode action.Condition.Expression as condition.expression" | when "action.Condition != nil"
    action "Omit the entire condition sub-table" | unless "action.Condition != nil"

This is Butter describing itself — the same extension that produces this TOML output is documented and defined in a .butter spec. Now let's build the extension that makes it work.

Step 1: Create the package

mkdir -p pkg/output/toml

Step 2: Implement the Extension

// pkg/output/toml/toml.go
package toml

import (
    "bytes"
    "fmt"

    "butter/pkg/ast"
    "butter/pkg/output"

    "github.com/BurntSushi/toml"
)

func init() {
    output.Register(tomlExt{})
}

type tomlExt struct{}

func (tomlExt) Name() string          { return "toml" }
func (tomlExt) FileExtension() string { return ".toml" }

func (tomlExt) Serialize(spec *ast.AppSpec) ([]byte, error) {
    var buf bytes.Buffer
    enc := toml.NewEncoder(&buf)
    if err := enc.Encode(spec); err != nil {
        return nil, fmt.Errorf("toml encode: %w", err)
    }
    return buf.Bytes(), nil
}

Key points:

  • The init() function calls output.Register(). This is automatic — importing the package is enough to register it.
  • Name() returns "toml" — this is the value the user passes to --format toml.
  • FileExtension() returns ".toml" — used for the default output filename.
  • Serialize() uses the TOML library's encoder. Since AST types have toml struct tags, toml.Encode works directly.
Note on struct tags: AST types currently have json and yaml tags. If your format uses different tag keys (e.g. toml), you can either add the tags to pkg/ast/types.go or implement Marshaler interfaces for custom serialisation.

Step 3: Add the dependency

go get github.com/BurntSushi/toml

Step 4: Register in the CLI

Add a blank import to cmd/root.go:

import (
    _ "butter/pkg/output/htmltree" // <-- experimental
    _ "butter/pkg/output/json"
    _ "butter/pkg/output/toml"     // <-- new
    _ "butter/pkg/output/yaml"
)

Step 5: Build and use

go build -o butter main.go
butter compile demo.butter -f toml -o demo.toml

The --format flag description and error messages automatically reflect the new extension:

$ butter compile --help
...
  -f, --format string   Output format (default: json). Supported: json, toml, yaml

$ butter compile demo.butter -f toml
Successfully compiled: demo.butter ──> demo.toml

How Registration Works

The registry is a simple map[string]Extension in pkg/output/extension.go:

var registry = map[string]Extension{}

func Register(ext Extension) {
    name := ext.Name()
    if _, ok := registry[name]; ok {
        panic(fmt.Sprintf("output extension %q already registered", name))
    }
    registry[name] = ext
}

func Get(name string) (Extension, bool) {
    ext, ok := registry[name]
    return ext, ok
}

func Names() []string {
    names := make([]string, 0, len(registry))
    for n := range registry {
        names = append(names, n)
    }
    sort.Strings(names)
    return names
}

Go's init() ordering guarantees that all imported extension packages register themselves before cmd's init() runs, so the flag description is always up to date.

CLI Integration

The compile command's output stage was refactored from a hardcoded switch to a registry lookup:

extIface, ok := output.Get(outputFormat)
if !ok {
    return fmt.Errorf("unsupported output format %q — supported: %s",
        outputFormat, formatList())
}
outputData, err := extIface.Serialize(appAST)
ext := extIface.FileExtension()

This means every registered extension:

  • Appears in --format help text automatically
  • Is listed in error messages for invalid formats
  • Gets its FileExtension used for default output filenames
  • Works with --check (validation only, no serialisation)

Existing Extensions

The two built-in extensions live in the pkg/output/ directory and serve as reference implementations:

PackageNameExtensionLibrary
pkg/output/json/json.go "json" .json encoding/json (stdlib)
pkg/output/yaml/yaml.go "yaml" .yaml gopkg.in/yaml.v3
pkg/output/htmltree/htmltree.go "htmltree" .html html/template (stdlib)

Each is roughly 20 lines: a struct, three methods, and an init() registration call. The htmltree extension is an experimental format that renders the AST as an interactive, zoomable HTML tree — see it in action with butter compile spec.butter -f htmltree -o spec.html.

Publishing Your Extension

Extensions are regular Go packages — publish them like any other Go library:

  1. Publish to GitHub/GitLab — Push your package to a repository. The package should be importable as github.com/you/butter-formatname.
  2. Document the dependency — Tell users to go get your package and add the blank import to cmd/root.go (or main.go).
  3. Name conventions — Use a descriptive module name like butter-toml or butter-protobuf. Keep the package simple — one init() registration, one Extension implementation.
Tip: Extensions are compiled into the binary. There is no runtime extension loading, no Wasm, no shared libraries. This keeps the compiler a single static binary while maintaining zero-core-dependency extensibility.
