Feature

The primary organisational unit in Butter. A feature declares a discrete capability — a module, API, or sub-system — with typed parameters, sequential actions, and conditional execution logic.

A feature block describes what to build and how to build it. It declares typed input parameters, a sequential list of execution steps (actions), and optional conditions that decide which actions run. When compiled to JSON and fed to an AI agent, the feature gives the agent a complete execution model — not just a data schema.

Features are self-contained. All parameters, actions, and conditions are scoped entirely within the feature block. There are no cross-feature references, no shared state, no global side effects. This isolation means each feature can be compiled, tested, and reasoned about independently.

Syntax

A feature is declared at the top level of a .butter file, directly under the app / product header:

app MyApp
description "My application"
version "1.0.0"

feature FeatureName
  description "What this feature does"
  version "1.0.0"

  params
    param ParamName
      type string
      required true

  actions
    action "First step"
    action "Second step" | if "ParamName is set"

Rules

  • Each feature must have a unique name within the spec — duplicate feature names produce a semantic error.
  • The feature name is an identifier: letters, digits, underscores, and hyphens are allowed.
  • The content under a feature must be indented (spaces or tabs — tabs normalise to 4 spaces).
  • Valid keywords inside a feature: description, version, params, actions.
  • A feature can have zero params and zero actions — an empty feature is valid.
Note: The endpoint primitive shares the same top-level context as feature but adds HTTP-specific fields (route, method, responses, returns). See the Endpoint documentation for details.

Sub-blocks

A feature contains two optional sub-blocks, each declared with a container keyword followed by an indented block of items:

ContainerItem keywordPurpose

feature ProcessPayment

  version "1.0.0"

  params
    param OrderID
      type string
      required true
    param Amount
      type float
      required true

  actions
    action "Validate routing balance metrics"
    action "Apply cryptocurrency surcharge" | when "PaymentMethod is Crypto"
    action "Flag for manual review" | if "Amount > 10000"

Params

The params block declares the inputs a feature accepts. Each param is a named variable with a type and optional constraints. Parameters define the boundary between what the caller provides and what the feature's actions operate on.

params
  param Title
    type string
    required true
  param Priority
    type enum["low", "medium", "high", "urgent"]
    default "medium"
  param Limit
    type int
    default 50
    validate ">=1"
    validate "<=200"

Implicit Defaults

  • A param without an explicit type defaults to string.
  • A param without an explicit required defaults to false.

Semantic Rules

  • Parameter names must be unique within a feature — duplicates produce a semantic error.
  • A parameter marked required true with a default triggers a warning (the default is never used).
  • validate rules require type int or type float.
  • length and validate are mutually exclusive on the same parameter.

Actions

The actions block declares the sequential execution steps that an AI agent must perform. Actions are the core of Butter's intent model — they tell the AI what to do, in what order, and under what conditions.

actions
  action "Validate title is not empty"
  action "Assign unique identifier to the new task"
  action "Set creation timestamp to current time"
  action "Send notification to assigned user" | if "Priority == urgent"
  action "Schedule reminder for due date" | when "DueDate is not empty"

Execution Model

  • Sequential — Actions run one after another in the order they are declared. Never in parallel.
  • Deterministic — Given the same inputs, the same actions will execute in the same order.
  • Conditional — An action with a condition only runs when the condition evaluates to true.
  • Atomic — Each action is a single step. Complex operations should be broken into multiple actions.

Statement Format

The action statement is a quoted string describing what to do. It should be written as an imperative instruction:

action "Validate the checkout token is active"
action "Calculate the order total with applicable discounts"
action "Process the payment transaction"
action "Log the transaction audit trail"
Writing tip: Write action statements as if you are instructing a developer. Be specific about what to validate, calculate, or create. Vague actions like "handle the data" produce vague implementations.

Conditions

Actions can have optional conditions attached via the pipe (|) character. A condition determines whether the action runs based on the current parameter values or external state.

action "Process payment" | if "Amount > 0"
action "Skip validation" | unless "TrustedSource == true"
action "Notify admin" | when "ThresholdExceeded"
action "Keep alive" | while "ConnectionActive"
KeywordMeaningExample
ifExecutes only if the predicate evaluates to true| if "Amount > 10000"
unlessExecutes except when the predicate evaluates to true| unless "dry_run == true"
whenReactive or event-driven hook; triggers upon an external event or state shift| when "DueDate is not empty"
whileActive polling or operational state persistence; the condition must remain continuously active| while "ConnectionActive"

Condition Expressions

The expression inside the quotes is a natural-language predicate. The AI agent interprets it against the available parameters and context. Common patterns:

  • Equality: "Status == active", "Status != archived"
  • Numeric comparison: "Amount > 10000", "Limit <= 200"
  • Existence: "CouponCode is set", "DueDate is not empty"
  • Boolean: "dry_run == true", "TrustedSource"
Note: Conditions are written in natural language inside quotes. The AI agent interprets them — there is no formal expression evaluator. Keep expressions clear and unambiguous for best results.

Enforce Rules

Actions can have zero or more enforce rules as indented children. An enforce rule declares a condition that must hold for the action to succeed — it is a constraint, not a conditional branch.

action "Validate routing balance metrics"
  enforce "The payment gateway must have sufficient routing capacity"
  enforce "Failed validations must log the routing error before halting"

Enforce vs Condition

AspectCondition (| if)Enforce
PurposeDecides whether the action runsDeclares a constraint that must hold during execution

Writing Enforce Rules

Write enforce rules as invariants — things that must always be true for the action to succeed. They are instructions to the AI agent about validation boundaries:

action "Charge customer payment method"
  enforce "The charge amount must not exceed the order total"
  enforce "The payment method must be verified before charging"
  enforce "A receipt must be generated after successful charge"

Scope Isolation

Every feature in Butter is strictly self-contained. Parameters, actions, and conditions within a feature have no knowledge of or dependency on other features. This design has critical implications:

  • No cross-references — A feature cannot reference a parameter from another feature.
  • No shared state — Features are independent execution models. One feature's actions do not affect another's.
  • Independent compilation — Each feature can be isolated, compiled, and reasoned about on its own.
  • Zero noise — When you extract a single feature, the output contains zero references to unrelated code.

This isolation is what makes selective compilation possible. Because each feature is a complete, self-contained unit, the compiler can extract it without resolving external dependencies or performing cross-file linking.

feature CreateTask
  params
    param Title
      type string
      required true
  actions
    action "Validate title is not empty"
    action "Assign unique identifier"

feature DeleteTask
  params
    param TaskID
      type string
      required true
  actions
    action "Verify TaskID exists"
    action "Remove task record"

Compiling with --only feature:CreateTask produces output containing only CreateTask — the AI agent sees zero mention of DeleteTask, its parameters, or its actions.

Selective Compilation

Butter's CLI supports extracting specific features via the --only and --exclude flags. This is the primary mechanism for building targeted AI prompts.

Extract a Single Feature

butter compile app.butter --only feature:CreateTask -f json

Extract Multiple Features

butter compile app.butter --only feature:CreateTask,feature:DeleteTask -f json

Exclude a Feature

butter compile app.butter --exclude feature:DeleteTask -f json

How It Works

  1. The compiler parses the full .butter file into a complete AST.
  2. Semantic analysis runs on the full AST (all features are validated).
  3. The filter engine prunes the AST — un-matched feature branches are dropped.
  4. The global context (app name, description, version) is always preserved.
  5. The isolated sub-graph is serialised to the requested format.
Error handling: If --only references a feature name that does not exist in the spec, compilation fails with an error. If --exclude references a non-existent feature, it also fails. The compiler never silently ignores typos.

Human-to-AI Handoff

Selective compilation is designed for the prompt-stacking workflow:

  1. Write the feature spec inside your .butter file.
  2. Extract with butter compile app.butter --only feature:CreateTask -f json -o prompt.json.
  3. Paste the JSON into your AI chat alongside instructions:
Implement the code for the following Butter contract definition.
Actions are sequential execution steps — run them in the listed order.
Respect all conditions, types, and constraints.

[Paste JSON here]

Compiled Output

When compiled to JSON, a feature produces a structured object with all its parameters, actions, conditions, and enforce rules:

feature ProcessPayment
  description "Processes financial transactions"
  version "1.0.0"

  params
    param OrderID
      type string
      required true
    param Amount
      type float
      required true

  actions
    action "Validate routing balance metrics"
      enforce "The payment gateway must have sufficient routing capacity"
    action "Apply cryptocurrency surcharge" | when "PaymentMethod is Crypto"
    action "Flag for manual review" | if "Amount > 10000"

Compiles to:

{
  "name": "ProcessPayment",
  "description": "Processes financial transactions",
  "version": "1.0.0",
  "params": [
    {
      "name": "OrderID",
      "type": "string",
      "required": true
    },
    {
      "name": "Amount",
      "type": "float",
      "required": true
    }
  ],
  "actions": [
    {
      "statement": "Validate routing balance metrics",
      "enforce": [
        "The payment gateway must have sufficient routing capacity"
      ]
    },
    {
      "statement": "Apply cryptocurrency surcharge",
      "condition": {
        "type": "when",
        "expression": "PaymentMethod is Crypto"
      }
    },
    {
      "statement": "Flag for manual review",
      "condition": {
        "type": "if",
        "expression": "Amount > 10000"
      }
    }
  ]
}

Examples

Minimal Feature

feature Greet
  params
    param Name
      type string
      required true
      default "World"
  actions
    action "Say hello to the user"

Feature with Multiple Conditionals

feature NotifyUser
  description "Sends notifications based on priority and channel preference"
  version "1.0.0"

  params
    param Recipient
      type string
      required true
    param Priority
      type enum["low", "medium", "high", "urgent"]
      default "medium"
    param Channel
      type enum["email", "sms", "push"]
      default "email"

  actions
    action "Validate recipient contact information"
    action "Queue message for delivery"
    action "Mark as high-priority" | if "Priority == urgent"
    action "Escalate to on-call manager" | if "Priority == urgent"
    action "Send via SMS gateway" | when "Channel is sms"
    action "Send via push notification" | when "Channel is push"
    action "Retry delivery on failure" | while "Delivery attempt fails"

Feature with Enforce Rules

feature ChargePayment
  description "Processes payment charges with safety constraints"
  version "1.0.0"

  params
    param Amount
      type float
      required true
      validate ">0"
    param Currency
      type string
      default "USD"

  actions
    action "Verify sufficient account balance"
      enforce "Balance must exceed charge amount by at least 10%"
      enforce "Account must not be frozen or suspended"
    action "Apply currency conversion rate"
    action "Submit charge to payment processor"
      enforce "Charge must include idempotency key"
      enforce "Processor response must be logged"
