Output Extensions Reference

Every built-in output format documented in detail — purpose, implementation, output examples, and when to use each one.

Butter's output layer is built around a pluggable Extension interface. Each format is a standalone Go package that registers itself with the compiler at startup. The --format (or -f) flag selects which extension to use at compile time.

Three built-in extensions ship with the compiler:

NameFile ExtensionPrimary Use Case
json.jsonStructured data for AI agents, CI pipelines, and programmatic consumption
yaml.yamlHuman-readable configuration and documentation
htmltree.htmlInteractive visual debugging and spec exploration in the browser

All extensions are imported via blank imports in cmd/root.go and auto-discovered at runtime. The compiler lists them automatically in --help output and error messages.

JSON Extension

PropertyValue
Packagepkg/output/json/json.go
Extension namejson
File extension.json
Libraryencoding/json (Go standard library)
Default format?Yes — butter compile uses JSON when no -f flag is given

Purpose

JSON is the primary output format for Butter. It produces a clean, pretty-printed JSON representation of the complete AST — every app, feature, param, action, condition, and enforce statement maps directly to a JSON object. This is the format designed to be consumed by AI agents: the structured tree tells the agent exactly what to build, in what order, and under what conditions.

JSON is also the default output format. Running butter compile spec.butter without any flags produces spec.json.

Source Code

package json

import (
    "encoding/json"
    "butter/pkg/ast"
    "butter/pkg/output"
)

func init() {
    output.Register(jsonExt{})
}

type jsonExt struct{}

func (jsonExt) Name() string          { return "json" }
func (jsonExt) FileExtension() string { return ".json" }

func (jsonExt) Serialize(spec *ast.AppSpec) ([]byte, error) {
    return json.MarshalIndent(spec, "", "  ")
}

The implementation is 21 lines. json.MarshalIndent serialises the entire AST with two-space indentation. The omitempty struct tags on Description, Version, Params, and Actions ensure that optional fields are omitted from the output when they are empty, keeping the JSON lean.

Example Output

}

When to Use

  • AI agent consumption — The structured format gives agents a complete, unambiguous specification tree.
  • CI/CD pipelines — Lightweight parsing, broad language support, easy to diff.
  • Programmatic introspection — jq, python -m json.tool, and every language's standard library can consume it.
  • Default workflow — If you don't have a specific reason to choose another format, use JSON.

YAML Extension

PropertyValue
Packagepkg/output/yaml/yaml.go
Extension nameyaml
File extension.yaml
Librarygopkg.in/yaml.v3 (third-party)

Purpose

YAML produces a human-readable representation of the same AST that JSON serialises. It is ideal for documentation, configuration management, and any workflow where people need to read the compiled spec directly. YAML's indentation-based structure mirrors the .butter source's own indentation style, making it the most visually natural alternative format.

Source Code

package yaml

import (
    "butter/pkg/ast"
    "butter/pkg/output"
    "gopkg.in/yaml.v3"
)

func init() {
    output.Register(yamlExt{})
}

type yamlExt struct{}

func (yamlExt) Name() string          { return "yaml" }
func (yamlExt) FileExtension() string { return ".yaml" }

func (yamlExt) Serialize(spec *ast.AppSpec) ([]byte, error) {
    return yaml.Marshal(spec)
}

The YAML extension is structurally identical to JSON's: 21 lines, a struct, three methods, and an init() registration. It delegates to the yaml.v3 marshaller, which uses the yaml struct tags on the AST types (yaml:"params,omitempty", etc.). Internal fields such as line numbers are tagged with yaml:"-" so they never appear in the output.

Example Output

app: TodoApp
description: A full-featured todo application...
version: "1.0.0"
features:
  - name: CreateTask
    description: Allows creation of new tasks...
    version: "1.0.0"
    params:
      - name: Title
        type: string
        required: true
      - name: Priority
        type: enum["low","medium","high","urgent"]
        default: medium
    actions:
      - statement: Validate title is not empty
      - statement: Assign unique identifier to the new task
      - statement: Send notification to assigned user
        condition:
          type: if
          expression: Priority == urgent

When to Use

  • Documentation — Include compiled YAML in README files or docs to show your project's spec in a human-friendly format.
  • Configuration pipelines — When the compiled output feeds into a YAML-based tool chain (e.g., Helm charts, GitHub Actions, Docker Compose).
  • Manual review — The indented format is easier to scan visually than JSON's brackets and quotes.
  • Git diffs — YAML's line-oriented structure produces cleaner diffs than JSON for the same data.
Note: YAML's yaml.v3 library is the only third-party dependency across all output extensions. JSON and htmltree use only Go standard library packages.

HTML Tree Extension

PropertyValue
Packagepkg/output/htmltree/htmltree.go
Extension namehtmltree
File extension.html
Libraryhtml/template (Go standard library) + inline HTML/CSS/JS
StatusExperimental

Purpose

The HTML tree extension renders the spec as a standalone, interactive web page. It transforms the AST into a visual tree of cards connected by animated SVG bezier curves. Unlike JSON and YAML (which are data formats), htmltree is a visualisation tool — it helps developers explore, debug, and present their Butter specifications in the browser.

The entire output is a single self-contained HTML file with embedded CSS and JavaScript. No external dependencies, no network requests, no server required. Open it directly from disk in any modern browser.

Source Code

The implementation lives in pkg/output/htmltree/htmltree.go (373 lines). The Go code is minimal — it marshals the AST to JSON, then injects the JSON into an HTML template. The heavy lifting happens in the embedded JavaScript that runs in the browser:

package htmltree

import (
    "bytes"
    "encoding/json"
    "html/template"
    "butter/pkg/ast"
    "butter/pkg/output"
)

func init() {
    output.Register(htmlTreeExt{})
}

type htmlTreeExt struct{}

func (htmlTreeExt) Name() string          { return "htmltree" }
func (htmlTreeExt) FileExtension() string { return ".html" }

func (htmlTreeExt) Serialize(spec *ast.AppSpec) ([]byte, error) {
    specJSON, err := json.Marshal(spec)
    if err != nil { return nil, err }
    var buf bytes.Buffer
    if err := tmpl.Execute(&buf, map[string]interface{}{
        "Spec": template.JS(specJSON),
        "App":  spec.App,
    }); err != nil { return nil, err }
    return buf.Bytes(), nil
}

Visual Features

Three-Column Layout

The tree is organised into three visual columns from left to right: the root app node, the feature cards, and the params/actions detail panels. SVG bezier curves connect nodes across columns: the app node to each feature, and each feature to its params and actions sub-panels.

Interactive Controls

  • Collapse/Expand — Features start collapsed. Click the +/− toggle next to features on the app card to expand or collapse all features. Individual params and actions panels toggle independently per feature.
  • Zoom — Scroll the mouse wheel to zoom in and out. Zoom is centred on the cursor position.
  • Pan — Click and drag the background to pan the view.
  • Auto-fit — The view automatically centres and scales to fit all visible cards on initial load, after every toggle, and on window resize.

Card Details

Each node card displays structured information about its corresponding AST node:

  • Title bar — Blue accent with the node name and a version badge.
  • Description — Keyword on its own row, value on an indented second row (separate backgrounds for readability).
  • Version — Inline key-value row.
  • Params — Each param is two rows: param name (purple accent) and its type/default/constraint info (muted text).
  • Actions — Numbered sequentially (1., 2., ...). Conditions (if/unless/when/while) and enforce statements render as separate indented rows with a left border bracket linking them to their parent action. Distinct background colours differentiate condition rows from enforce rows.

Example Output

Run butter compile todo.butter -f htmltree -o todo.html and open todo.html in your browser. The generated page is fully self-contained — no server, no internet connection needed.

The root card shows the app name (TodoApp) with its description and version. Below it, a toggle row labelled features: [4 items] (collapsed by default) with a + icon. Click the toggle to expand: feature cards slide into column two, and bezier curves draw from the app card to each feature card. Each feature card has its own description, version, and two sub-toggles for params and actions. Clicking those reveals the detail panels in column three with param rows, numbered action statements, and indented condition/enforce rows.

When to Use

  • Spec exploration — When you want to visually inspect a spec, seeing the full tree structure at a glance.
  • Debugging — Quickly verify that the compiler parsed your .butter file correctly by examining the visual tree.
  • Presentations and demos — Open a spec in the browser during code reviews or team meetings.
  • Teaching — The visual layout makes it easy to explain how Butter's app→feature→action hierarchy works.
Note: The htmltree extension is marked experimental. Its output format may change across compiler versions as visual features are added. Always regenerate HTML output with the current compiler version for accurate results.

For most workflows, JSON is the right choice — it is the default, has zero dependencies, and is designed for AI agent consumption. Use YAML when you need human readability alongside machine parseability. Use htmltree when you want to see your spec come to life as an interactive diagram.
