v1.13.0 — 2026-07-13

  • endpoint enhancement — New top-level block for defining synchronous HTTP transport network architecture alongside features. Supports version, params, responses, actions, and returns sub-blocks with strict scope separation from features.
  • response blocks — Define reusable response schemas with typed fields inside endpoints. Fields support recursive nested sub-fields.
  • return mapping — Map status codes to response references or string payloads with optional if/unless conditions.
  • field keyword — Typed fields inside responses with optional nested sub-fields; defaults to string when no type is specified.
  • Return statement coloring — Status codes, response references, and string payloads in return statements are syntax-highlighted with dedicated TextMate patterns.
  • Type value coloring — Values after type keywords now receive support.type.butter scope.
  • Response ref hover — Hovering over a response reference in a return statement shows the response schema declaration.
  • Response ref navigation — Ctrl+click on a response reference in a return statement navigates to its response declaration.
  • HTML tree visualization — Endpoints render as node cards with route/method badges, response schema cards, and return mapping cards with connection lines.
  • AI simulator — Endpoints appear in the sidebar with route/method display, parameter forms, response schemas, and return mapping.
  • Formatter — Handles endpoint, responses, returns, response, and field keywords with proper indentation rules.

v1.12.0 — 2026-07-11

  • Prompt output extension — New -f prompt output format generates a token-efficient Markdown spec designed for LLM consumption. Features # [SYSTEM SPEC] headers, ## Feature: sections with versioned params (type, required/optional, defaults, validate constraints), and numbered ### Execution Sequence steps with ENFORCE, WHEN, IF, UNLESS, and WHILE condition sub-bullets. Output defaults to .prompt.md.

v1.11.2 — 2026-07-09

  • Fixed raw newlines and tabs in compiled output — String values (descriptions, defaults, action statements) no longer contain literal \n or \t characters in JSON, YAML, or other output formats. These were leaking into compiled output when descriptions were written across multiple lines for readability in .butter source files. The lexer now properly handles escape sequences (\", \\, \n, \t) and strips control characters from string tokens.

v1.11.1 — 2026-07-08

  • Fixed provider prompt UX — Instead of requiring the user to type opencode or paste an API key into a blank input, the sim extension now presents a numbered menu of all providers. Selecting by number works immediately; API keys are only requested for providers that need one (Anthropic, ChatGPT, Gemini, DeepSeek). Selecting opencode or none proceeds straight to code generation without any key prompt.
  • Fixed param-driven action logic — The AI code generation prompt now explicitly instructs models to always evaluate conditions using the params object, ensuring actions behave correctly when params are defined in the spec.
  • Fixed version badges across all documentation — All doc pages now consistently show the correct current version.

v1.11.0 — 2026-07-06

  • Sim extension with AI-powered spec simulation — The new sim output format (-f sim) generates a self-contained HTML page for interactively testing spec logic. Features render as typed input controls; the Run Simulation button evaluates conditions and animates action results. AI-generated code executes directly in the browser via AISim.run() and drives the action flow with green/red/amber status cards. Built-in JS expression evaluator handles conditions without any AI.
  • opencode provider — BUTTER_AI_PROVIDER=opencode uses the opencode CLI with free built-in models (default opencode/deepseek-v4-flash-free, override via BUTTER_AI_MODEL). No API key required — pass opencode at the key prompt or set BUTTER_AI_API_KEY=opencode. The opencode CLI must be installed separately.
  • Provider prompt with "none" fallback — The interactive prompt now lists all provider options (anthropic/chatgpt/gemini/deepseek/opencode/none). Selecting none skips AI code generation entirely and uses the built-in evaluator.
  • AI output validation — Generated code is validated by isValidAICode() before embedding. Natural language responses (which don't contain AISim.run or AISim =) are rejected with a warning and fall back to the built-in evaluator.
  • Disclosure triangle icon fix — The "AI Source Code" collapsible panel previously had a broken disclosure icon due to html/template CSS escaping. The icon is now rendered as an inline <span> inside <summary> with list-style: none.
  • Shared stdin reader — Replaced multiple bufio.Scanner instances with a shared bufio.NewReader + readLine() to fix buffered data loss between the key and provider prompts.
  • Rate limit retry — API calls retry up to 2 times with exponential backoff (2s, 4s) on 429 responses. Error messages now extract error.message from the API JSON response instead of dumping the full body.

v1.10.1 — 2026-07-04

  • Mobile responsive htmltree — The htmltree output format now works on mobile devices with touch pan, pinch zoom, and double-tap zoom. The 3-column tree layout is preserved on all screen sizes. SVG bezier curves are no longer drawn when features are collapsed.

v1.10.0 — 2026-07-03

  • Interactive HTML tree visualizer — The htmltree output format now produces a zoomable, interactive SVG tree with bezier-curve connections between app, features, params, and actions. Long text wraps within fixed-width cards. Initial view collapses features; click to expand.
  • Stacked detail rows — Description, param, condition, and enforce fields render as keyword row + value row with distinct backgrounds, improving readability.
  • Visual grouping — Conditions and enforce statements are visually linked to their parent action via a left border bracket.
  • Action numbering — Actions prefixed with sequential numbers (1., 2., ...).
  • Conditional and enforce rows — if/unless/when/while and enforce statements now render on separate indented rows with distinct background colors.
  • Defensive null guards — All .forEach iterations guarded with ||[] to handle features without params or actions.

v1.9.0 — 2026-07-02

  • IntelliSense code completion — Context-aware autocomplete for Butter keywords, types, parameter fields, and conditionals. Completion analyses indentation and parent block context to suggest only valid keywords for the current position. Snippets for app, product, feature, param, action, enum[...].
  • Hover documentation — Hover over any Butter keyword, type, or conditional to see a markdown description with syntax examples.
  • Refined philosophy — All documentation now consistently communicates Butter's purpose: a specification language that communicates intent to AI agents for up to 100% one-shot results. The todo.butter example uses app instead of product and is fully aligned with the working todo.html demo.

v1.8.0 — 2026-07-01

  • Action enforce field — Actions now support zero or more enforce strings as indented children, specifying conditions that must hold for the action to succeed. Stored as Enforce []string in the AST.
  • Updated syntax highlighting — enforce added to TextMate grammar (VS Code) and Prism (docs) keyword lists. Token color customizations match docs: amber keywords, green strings, blue functions, purple variables.
  • Compiler spec alignment — compiler.butter and docs/compiler-spec.html synced to v1.8.0 with enforce examples.
  • Updated extension — butter-extension v1.8.0 with enforce highlighting and color customizations.

v1.7.0 — 2026-06-30

  • Pluggable output extensions — Output formats are now registered via a Go Extension interface. Writing a new format is a single-file, three-method implementation.
  • Extension registry — pkg/output.Extension interface with Register()/Get()/Names(). Extensions auto-register via init().
  • JSON and YAML refactored as extensions — Moved from hardcoded functions in parser.go to standalone extension packages.
  • Dynamic help text — --format flag description and error messages automatically list all registered extensions.
  • Cleaner YAML output — Internal line numbers no longer leak into YAML output (yaml:"-" tags added).
  • Developer documentation — New Writing Extensions guide with a complete TOML extension walkthrough.

v1.6.0 — 2026-06-29

  • Semantic analysis pass — Catches duplicate feature/param names, type-default mismatches, enum defaults, and redundant required+default combos.
  • Enhanced --check — butter compile --check now validates semantics in addition to syntax.
  • Source location tracking — AST nodes carry precise line numbers for accurate error reporting.

v1.5.0 — 2026-06-28

  • YAML output — New --format / -f flag to generate YAML output (butter compile demo.butter -f yaml). JSON remains the default.
  • Updated philosophy — Clarified that core compilation (lexer, parser, validator) is zero-dependency, while output serialization may leverage libraries for format flexibility.

v1.4.0 — 2026-06-25

  • bool type — New bool parameter type for boolean values (true / false).
  • length field — New length parameter field for exact length constraints (e.g. length 13). Mutually exclusive with validate.
  • Updated error messages — Param field error lists include validate and length; length/validate mutual exclusion enforced.

v1.3.0 — 2026-06-25

  • product keyword alias — product can now be used interchangeably with app at the top level, across parser, formatter, and syntax highlighting.
  • Numeric literals in defaults — Values like default 50 now parse correctly. The lexer now allows digits as identifier start characters.
  • Parser robustness — parseParam loop now guards against infinite loops on unexpected token types.
  • todo.butter example — New complete todo application example using product with four features, enum types, and integer defaults.
  • Validate rule format validation — validate rules are now rejected at parse time if they don't match a valid numeric comparison (operator + number). Supported operators: >, >=, <, <=, =<, ==, =, !=.
  • Validate type restriction — validate rules are now rejected if the parameter type is not int or float.

v1.2.0 — 2026-06-24

  • Butter Format (butter fmt) — New subcommand to automatically format .butter files. Uses a two-pass algorithm: removes blank lines after parameter keywords, then inserts blank lines before params, actions, and between top-level feature blocks.
  • Format on save — The VS Code extension now automatically runs butter fmt on every save, no configuration needed.
  • Butter: Format current file command — Available in the VS Code command palette.

v1.1.0 — 2026-06-23

  • --check flag — butter compile --check validates syntax without writing a .json file. Returns exit code 0 on success, 1 on error.
  • On-save linting — The VS Code extension runs butter compile --check on every save, surfacing errors with red squiggly underlines.
  • Butter: Lint current file command — Available in the VS Code command palette.
  • Configurable compiler path — butter.compilerPath setting lets you specify a custom path to the butter binary.
  • butter --version — Prints the compiler version.
  • Named capture highlighting — TextMate grammar now highlights app, feature, and param identifiers distinctly.
  • version keyword — Added to syntax highlighting.
  • Document icon — Custom SVG icon for .butter files.

v1.0.0 — 2026-06-22

  • Initial release of the Butter DSL compiler.
  • butter compile command compiles .butter files to pretty-printed JSON.
  • Lexer with Python-style indentation tracking (INDENT/DEDENT tokens).
  • Recursive descent parser supporting app, description, version, feature, params, param (with type, required, default), actions, action, and pipe-delimited conditionals (if, unless, when, while).
  • VS Code extension with syntax highlighting, indentation rules, and comment toggling.
